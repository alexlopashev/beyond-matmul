\begin{abstract}
Modern machine-learning systems routinely lower semantically rich computations
into dense matrix multiplication because dense GEMM is universal, convenient,
and heavily optimized. That default is useful, but it can erase why a matrix
exists: a convolution becomes an anonymous Toeplitz matrix, an adapter becomes a
materialized product, an embedding projection becomes a dense multiply, and a
quantized operator becomes dequantized storage. This paper explores a
provenance-aware framing for fixed-weight inference in which dense GEMM remains
a valid fallback rather than the default semantic intermediate representation.
We propose a linear and affine operator IR that preserves or recovers structure,
a planner that selects exact or bounded-error lowerings under backend and reuse
contracts, and an evaluation program that ties frontend coverage, tests,
benchmarks, and case studies to the research claims.
\end{abstract}

\section{North Star}

The north star of Beyond Matmul is a completed research artifact with two
mutually reinforcing outputs:

\begin{enumerate}
  \item A final whitepaper that explains, evaluates, and bounds the claim that
  provenance-aware computation can improve fixed-weight ML inference by
  retaining structure normally lost during dense lowering.
  \item An accompanying codebase that provides executable evidence: operator
  semantics, Torch frontend capture, recovery probes, approximation builders,
  planner decisions, tests, demos, and benchmark data.
\end{enumerate}

The project is complete when the paper's claims are supported by the codebase
and its generated evidence, the GitHub wiki concisely defines the project for
humans and agents, and GitHub issues no longer contain unresolved high-priority
gaps against the stated research questions.

\section{Problem}

Dense matrix multiplication is an excellent implementation target and a robust
fallback abstraction. It is not always an excellent semantic representation.
When a framework lowers a structured computation into an anonymous matrix, it
often discards facts that matter for inference:

\begin{itemize}
  \item convolution preserves local receptive fields and weight sharing;
  \item low-rank adapters preserve factorization and rank bounds;
  \item embeddings preserve lookup semantics and repeated token structure;
  \item masks preserve causal, banded, block, or page sparsity;
  \item quantized weights preserve packing, scale, zero point, and codebooks;
  \item diagonal gates preserve elementwise scaling rather than general mixing.
\end{itemize}

Once these facts are erased, a downstream compiler or runtime must either use
dense GEMM or rediscover structure heuristically from the dense matrix. The
first path can waste work and memory; the second path is lossy and expensive to
trust.

\section{Thesis}

Many matrix multiplications in ML systems are semantically structured
computations that were densified for convenience. A provenance-aware
linear-operator IR can preserve or recover that structure and allow a planner to
choose exact or bounded-error lowerings that are cheaper, clearer, or more
controllable than dense GEMM for fixed-weight inference.

The claim is deliberately not ``GEMM is bad.'' Dense GEMM should remain one
lowering in a richer operator space. The research question is when and how much
additional provenance lets the system choose something better.

\section{Scope}

The initial artifact focuses on fixed-weight inference. This scope makes reuse
contracts explicit: preprocessing can be amortized across calls, weights do not
mutate during training, and exact dense equivalence can be checked for small
operators.

Out of scope for the first complete artifact:

\begin{itemize}
  \item training-time weight mutation and gradient semantics;
  \item production GPU kernels;
  \item claims of hardware-calibrated speedup without benchmark evidence;
  \item universal Torch operation support in a single pass.
\end{itemize}

\section{Computation Reason Provenance}

This project uses provenance to mean the reason an operator exists, not merely
where the tensor was allocated. A provenance-aware operator records:

\begin{itemize}
  \item source framework node or expression;
  \item input and weight identities;
  \item transformation history, such as convolution to Toeplitz or adapter
  factors to a merged dense weight;
  \item structural metadata, such as rank, sparsity, kernel shape, codebook, or
  quantization layout;
  \item exactness or approximation contract;
  \item reuse and cache assumptions;
  \item backend and layout constraints.
\end{itemize}

This framing lets later passes ask not only ``what matrix is this?'' but ``what
computation did this matrix come from, and what lowerings are still valid?''

\section{System Design}

The artifact is organized around six components.

\subsection{Frontend Capture}

The first method is to capture fixed-weight computation before the framework
has lowered it to an anonymous dense product. The current Torch FX frontend
walks graph nodes, resolves fixed parameters, buffers, constants, and selected
module attributes, then emits executable IR operators whose provenance records
the source node, expression, transform history, and captured structure. This is
a provenance-preserved path: the structured operator is produced from the
original computation rather than inferred from a dense matrix after the fact.

The supported surface is intentionally narrow and test-backed. The frontend
currently captures nested \texttt{nn.Linear} or \texttt{F.linear} factors,
named adapter pairs, embedding followed by projection, single-channel and
multi-channel \texttt{Conv1d} including grouped/depthwise and scalar
stride/padding/dilation variants, functional \texttt{conv1d}, fixed-weight
\texttt{matmul}/\texttt{mm} forms, fixed-bias \texttt{addmm}, and exported
graph fixed-weight \texttt{addmm}/nested-linear recovery through graph
signature state. It does not claim \texttt{Conv2d}, quantized linear or
convolutional modules, non-scalar Conv1d parameters, invalid Conv1d output
lengths, non-default \texttt{addmm} scaling, shape-inferred matmul orientation,
or dynamic-weight matmul/addmm. The coverage matrix currently has supported
and unsupported rows, with no intermediate \texttt{Next} rows; broader exported
graph variants and quantized operators remain future work until represented and
tested. The evidence anchor is the \texttt{Torch frontend coverage} row in
\texttt{docs/evidence\_matrix.md}, with the detailed boundary maintained in
\texttt{docs/torch\_frontend\_coverage.md}.

\subsection{Operator IR and Contracts}

The IR represents fixed-weight linear maps with shape
$(\mathrm{out\_features}, \mathrm{in\_features})$ and affine maps as a linear
operator plus bias. Applying a linear operator to a row-major batch computes
$xW^T$; applying an affine operator computes $xW^T + b$. Metadata stores
provenance, structural fields, approximation contract, quantization placeholder
data, reuse budget, layout, hardware target, and allowed lowerings. The
contract is part of the operator rather than an external note: exact operators
require backend and layout support, while approximate operators additionally
carry metric, epsilon, observed error, and sample-count evidence.

Implemented operator families include dense, diagonal, sparse COO, low-rank,
affine, codebook, bitpacked binary, single-channel Conv1d, and multi-channel
Conv1d operators. Conv1d operators keep stride, padding, dilation, output
length, and grouped/depthwise channel partition metadata so the planner can
distinguish direct convolution lowerings from dense materialization. Every
implemented exact family exposes executable semantics and a dense fallback,
which keeps dense GEMM available for equivalence checks and portability. The
evidence anchor is the \texttt{IR operator families} row in
\texttt{docs/evidence\_matrix.md}; \texttt{docs/ir\_spec.md} is the compact
schema reference.

\subsection{Recovery Analyzer}

When provenance is lost, cheap analyzers probe dense matrices for recoverable
structure. This is the dense-recovery path, and it is intentionally weaker
than frontend preservation. A recovered candidate may explain a matrix, but it
cannot prove the original computation reason unless separate provenance is
available. For that reason, analyzer output carries bounded heuristic
confidence, explicit probe details, and, when sample inputs are supplied,
output-relative validation evidence.

The current probes cover diagonal residuals, sparsity, block sparsity,
permutation-like structure, small codebooks, low-rank approximation, repeated
blocks, and repeated fixed-weight sightings. Confidence values are
probe-specific ranking scores, not calibrated probabilities. If a recovered
candidate is executable, validation on representative inputs records
\texttt{output\_relative\_l2}, sample count, sample exactness, and a
validation-derived confidence cap. Dense GEMM remains the exact fallback
whenever recovery is ambiguous or unsupported. The evidence anchor is the
\texttt{Recovery after lost provenance} row in
\texttt{docs/evidence\_matrix.md}.

\subsection{Approximation Builders}

Approximation builders produce candidate low-rank, sparse, codebook, or binary
operators from a dense reference. The method is deliberately candidate-based:
builders create possible structured operators, attach approximate contracts,
and leave acceptance to validation and planning rather than treating matrix
compression as sufficient evidence. Supported builders currently include
low-rank approximation, sparse top-k selection, codebook quantization, and
bitpacked binary approximation.

Approximate candidates are scored with product-aware metrics when
representative inputs are available. Matrix-relative Frobenius error remains
useful for reconstruction, but output-relative error can reject a candidate
that looks good as a matrix approximation and behaves poorly on the inference
inputs of interest. The current deterministic ablation is intentionally small:
one synthetic dense case shows low-rank and sparse top-k candidates passing a
matrix-relative threshold but failing the same output-relative threshold, while
codebook and bitpacked candidates fail both. This supports the need for
output-aware acceptance; it is not broad model-quality evidence. The evidence
anchor is the \texttt{Approximation and error contracts} row in
\texttt{docs/evidence\_matrix.md}.

\subsection{Planner Selection}

The planner selects a valid lowering under exactness, error, reuse, backend,
and layout contracts. It enumerates the original operator, valid structured or
approximate candidates, and a dense fallback, rejects options whose contracts
do not satisfy the request, estimates cost proxies for the remaining options,
and selects the lowest-cost valid plan. Lowering names include direct Conv1d
variants, low-rank product, sparse kernel, codebook kernel, bitpacked kernel,
and dense GEMM or dense GEMM with bias.

The request specifies exact-only or bounded-error behavior, epsilon and error
metric, representative sample inputs for product-aware scoring, expected call
count for reuse amortization, and backend/layout constraints. Planner costs are
currently estimates: operation count, memory movement, cache footprint,
preprocessing cost, and amortized preprocessing over repeated calls. The
planner-contract ablation is therefore contract evidence rather than hardware
performance evidence. It compares exact-only and bounded-error requests on the
same fixed-weight case, rejects and then accepts a preprocessing-heavy
low-rank lowering across its reuse threshold, and rejects an unsupported GPU
codebook lowering while preserving dense GEMM as a valid fallback. The evidence
anchor is the \texttt{Planner exactness and fallback} row in
\texttt{docs/evidence\_matrix.md}.

\subsection{Evidence Surface}

The codebase must produce evidence that the paper can cite: tests for semantics
and negative capture cases, workload case-study artifacts for adapter and
Conv1d narratives, benchmark artifacts for cost comparisons, an
approximation-error ablation artifact, and docs/wiki pages that describe the
current boundary of truth. The support material includes a concise evidence
matrix in \texttt{docs/evidence\_matrix.md} that maps major claims to tests,
demos, benchmark artifacts, or future-work boundaries.

\section{Research Questions}

\begin{enumerate}
  \item Which common fixed-weight Torch inference patterns can be captured
  exactly before they become anonymous dense matrices?
  \item When provenance is missing, which structures can be recovered cheaply
  enough to be useful, and how should uncertainty be represented?
  \item Under what reuse and backend contracts does a structured lowering beat
  dense GEMM according to operation, memory, preprocessing, or measured latency
  proxies?
  \item How should approximate lowerings be evaluated so output-level error is
  aligned with real inference behavior?
  \item What minimum wiki, issue, and automation loop keeps agentic development
  aligned with the research goal instead of accumulating unrelated code?
\end{enumerate}

\section{Evaluation Plan}

The evaluation should combine controlled synthetic cases and small real
workload case studies.

\subsection{Controlled Cases}

\begin{itemize}
  \item exact diagonal operators;
  \item exact sparse operators;
  \item exact low-rank operators;
  \item exact convolutional operators;
  \item small-codebook and bitpacked approximations;
  \item random dense fallback baselines.
\end{itemize}

\subsection{Workload Case Studies}

\begin{itemize}
  \item Conv1d module and functional Conv1d capture before dense
  materialization;
  \item LoRA or adapter weights merged into a dense projection;
  \item Conv/linear lowering from a broader CNN-style block;
  \item attention projection or masked attention block;
  \item embedding plus projection in a language-model-like block;
  \item quantized linear or convolutional modules once the IR can represent
  their contracts honestly.
\end{itemize}

\subsection{Metrics}

\begin{itemize}
  \item dense equivalence for exact lowerings;
  \item output-relative error for approximate lowerings;
  \item matrix-reconstruction versus output-error decisions for a bounded
  approximation ablation;
  \item operation-count proxy;
  \item memory-footprint and memory-movement proxy;
  \item preprocessing cost and calls required to amortize it;
  \item wall-clock latency where measurement is meaningful;
  \item coverage matrix rows connected to executable tests.
\end{itemize}

\subsection{Current Benchmark Results}

The current results are generated artifacts rather than hand-curated tables.
The main fixed-weight benchmark is regenerated with
\texttt{mise exec -- uv run python benchmarks/fixed\_weight.py --json-output
docs/results/fixed\_weight.json}; local CI also regenerates the workload
case-study, approximation-error, and planner-contract JSON artifacts documented
in \texttt{docs/benchmark\_artifacts.md}.

In the fixed-weight benchmark, the planner request allows approximate
lowerings with a maximum relative error of $0.05$ over a batch of $32$ and
$32$ requested calls. The current artifact selects exact specialized lowerings
for diagonal, sparse, low-rank, small-codebook, single-channel Conv1d, and
multi-channel Conv1d cases, and selects dense GEMM for the random dense
fallback case. All rows are valid and exact in this controlled benchmark. The
recorded fields include selected lowering, relative error, estimated apply
cost, estimated memory bytes, preprocessing cost, amortized preprocessing, and
pure-Python seconds per apply for dense and selected lowerings. For example,
the diagonal and sparse cases select \texttt{diagonal\_kernel} and
\texttt{sparse\_kernel} with zero output-relative error and no preprocessing,
while the codebook case records preprocessing work that is amortized over the
requested calls. The Conv1d rows select direct Conv1d lowerings with zero
output-relative error rather than first materializing an equivalent dense
matrix.

The workload case-study artifact is evidence for provenance preservation, not
timing performance. It records a merged LoRA-style adapter selecting
\texttt{low\_rank\_product\_bias}, an \texttt{nn.Conv1d} module selecting
\texttt{conv1d\_channel\_direct}, and a functional fixed-bias Conv1d selecting
\texttt{conv1d\_channel\_direct\_bias}. Each row records output-relative error
against the Torch workload, planner cost and memory proxies, dense fallback
fields, and an explicit \texttt{measured\_timing: false} boundary.

The approximation and planner-contract artifacts bound the negative claims. In
the approximation ablation, low-rank and sparse top-k candidates pass the
matrix-relative threshold but fail the output-relative threshold on the sampled
inputs, so the output-aware planner keeps dense GEMM for that case. In the
planner-contract ablation, exact-only planning keeps dense GEMM, bounded-error
planning can choose low-rank under the stated error contract, reuse sensitivity
changes whether preprocessing-heavy options are valid, and unsupported GPU
specialization falls back to dense GEMM. These are deterministic Python
evidence artifacts. They support claims about planner behavior, proxies, and
case coverage; they do not establish hardware-calibrated speedups or production
backend performance.

\section{Development Methodology}

All code work should be test-driven and issue-driven. Each issue should be
small enough for one PR, labeled for priority, risk, area, kind, and status, and
linked to blockers when necessary. Worker agents implement from fresh worktrees
based on latest main. Reviewer agents independently check correctness, local CI
parity, minimality, and north-star alignment before merging. Retrospective
agents periodically trim duplication, detect stuck PRs, and create corrective
issues.

\section{Current Risks}

\begin{itemize}
  \item Coverage drift: docs may claim support that tests do not prove.
  \item Benchmark overclaiming: pure-Python proxies may be mistaken for
  production performance evidence.
  \item Abstraction inflation: operator families may grow before their contracts
  are stable.
  \item Agent drift: autonomous workers may optimize for local issue completion
  instead of paper-quality evidence.
  \item Recovery ambiguity: dense structure probes may look convincing; current
  confidence is heuristic, and output-error validation covers only supplied
  representative samples.
\end{itemize}

\section{Completion Criteria}

The project reaches its north star when:

\begin{enumerate}
  \item the whitepaper is internally consistent and all major claims map to
  tests, demos, benchmark artifacts, or clearly labeled future work, with the
  current mapping recorded in \texttt{docs/evidence\_matrix.md};
  \item the codebase covers the selected high-leverage fixed-weight operator
  families with exact equivalence and negative tests;
  \item benchmark outputs are generated as machine-readable artifacts suitable
  for paper figures;
  \item the wiki concisely defines the project, north star, coverage boundary,
  issue loop, and artifact map;
  \item GitHub issues show no unresolved priority-zero or priority-one blockers
  against the paper thesis;
  \item retrospective review finds no major duplication, stale claims, or stuck
  PRs that would change the conclusions.
\end{enumerate}

\section{Conclusion}

Beyond Matmul treats dense multiplication as a powerful lowering, not the only
truth a compiler or runtime should remember. By preserving computation reason
provenance, the project aims to make structured inference paths measurable,
testable, and available to a planner. The final success condition is not a large
surface area of code; it is a clear research argument with enough executable
evidence to make the argument hard to dismiss.
